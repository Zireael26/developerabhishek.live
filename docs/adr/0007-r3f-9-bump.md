# ADR-0007 — @react-three/fiber 9 + drei 10 bump

**Status:** Accepted, 2026-04-21
**Context:** Phase 5.1a (plan §5.1a). Pre-requisite for the hero R3F canvas (5.1b) and the full Wanderer crane port (5.1c).
**Author:** Claude (Code), co-author to Abhishek Kaushik.

## Context

The scaffold shipped with `@react-three/fiber@^8.17.10` + `@react-three/drei@^9.114.0`. Both predate React 19's stable release. `pnpm install` consistently logged unmet peer warnings:

```
@react-three/fiber 8.18.0 unmet peer react@">=18.0.0": found 19.2.0
@react-spring/core 9.7.5 unmet peer react@"^16.8.0 || ^17.0.0 || ^18.0.0": found 19.2.0
```

React 19 hardened the type constraints on `ReactNode`, `Dispatch`, and the scheduling primitives; `react-reconciler` (the internal dependency R3F 8 uses) pins an older React contract. The warnings are benign today (R3F 8 runs on React 19 in practice), but any forward-looking Phase 5 canvas work will drag in reconciler type incompatibilities that surface as compile errors or runtime drift.

R3F 9.x and drei 10.x both declare `react@^19.0.0` peers, bundle the matching reconciler, and clean up the type surface. This is the right time to bump — **before** any Phase 5 canvas code is written against the old API.

## Decision

Bump to the latest majors:

- `@react-three/fiber`: `^8.17.10` → `^9.6.0`
- `@react-three/drei`: `^9.114.0` → `^10.7.7`
- `three`: stays at `^0.169.0` — R3F 9 + drei 10 both work with it; upgrading three to `^0.170` would require a second migration pass for drei's three-mesh-bvh subdep. Defer.

## Consequences

**Good:**
- React 19 peer warnings cleared (measured: `pnpm install` output no longer flags fiber/drei).
- Phase 5.1b + 5.1c can be written against the current R3F 9 API directly — no migration debt.
- `typecheck` / `lint` / `build` all clean after the bump (verified locally on `phase-5/r3f-drei-bump` before merge).

**Costs:**
- drei 10 renames a handful of components (e.g., `MeshReflectorMaterial` prop shape drift); no current code uses them, so net-zero for this slice. The Phase-5.1b/c ports will land directly on drei 10.
- Lockfile growth: five new transitive packages (drei 10's animation helpers).

**Risks (plan Risk R1):**
- R3F 9.x is still absorbing hotfixes as of April 2026. If a canvas regression surfaces during Phase 5.1b/c, pin back to R3F `8.18` via the isolation ADR-0007 follow-up; the pinning is not catastrophic because the canvas code is code-split + Suspense-gated (`next/dynamic({ ssr: false })`), so a rollback is localised to the scene files.

## Alternatives considered

**Stay on R3F 8.** Reject. The React 19 peer warnings will worsen once 5.1b ships R3F code that relies on React 19 features (`use()` hook, `cache`). Bumping later means re-doing any early 5.1b/c migration work.

**Pin Three to 0.170+.** Defer. No feature in 0.170 is load-bearing for the scenes described in `_reference/portfolio/`; drei 10 works fine on 0.169. If the Wanderer port (5.1c) needs a 0.170-only API, revisit.

**Use `motion/react` instead of framer-motion to drop the canvas-adjacent bundle.** Out of scope for this ADR; measured in Phase 5.1b when the real canvas code lands.

## Follow-ups

- Slice 5.1b: port `_reference/portfolio/index.html:1583–1816` (agent-graph scene) to R3F 9. Use the R3F 9 `Canvas` + `useFrame` + `useThree` API directly.
- Slice 5.1c: port `_reference/portfolio/companion.js` (Wanderer crane, 221 lines) to R3F 9. Custom geometry + pose-driven choreography.
- Post-Phase-5: audit drei 10 tree-shaking once the canvas ships; many of drei 10's helpers import three subsystems (bvh, postprocessing) that should stay out of the first-paint bundle.
- If 5.1b surfaces regressions: rollback path is `pnpm add @react-three/fiber@^8.18.0 @react-three/drei@^9.114.0` — canvas code is code-split, so the rollback doesn't block other work.
